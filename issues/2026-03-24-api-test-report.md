## 发现的问题

### 问题 1: limit 参数未限制最大值

**文档描述**：
> limit 每页数量，默认 50，最大 100

**实际行为**：
```bash
GET /api/v1/contents?limit=200
# 返回 200 条数据，而非限制为 100
```

**影响**：可能导致大内存消耗或性能问题。

**建议**：在 API 层添加 `min(limit, 100)` 限制。

---

### 问题 2: 无效游标未校验

**预期行为**：无效游标应返回错误。

**实际行为**：
```bash
GET /api/v1/contents?cursor=invalid_cursor
# 返回第一页数据，不报错
```

**影响**：可能导致数据重复或客户端逻辑混乱。

**建议**：校验游标格式，无效时返回 400 错误。

---

### 问题 3: 源 API 分页字段缺失

**内容 API 返回**：
```json
{
  "data": [...],
  "next_cursor": "...",
  "has_more": true,
  "count": 100
}
```

**源 API 返回**：
```json
{
  "data": [...],
  "has_more": null,
  "next_cursor": null
}
```

**影响**：API 行为不一致，源 API 无法分页。

**建议**：统一 API 响应格式，或文档说明源 API 不支持分页。

---

### 问题 4: source_id 筛选无法验证

**原因**：返回的内容不包含 `source` 或 `source_id` 字段。

**影响**：无法确认筛选是否正确工作。

**建议**：返回内容中添加来源信息。

---

### 问题 5: 响应结构与文档不一致

**文档描述**：
```json
{
  "data": [...],
  "meta": {
    "next_cursor": "...",
    "has_more": true
  }
}
```

**实际返回**：
```json
{
  "data": [...],
  "next_cursor": "...",
  "has_more": true,
  "count": 100,
  "server_timestamp": "..."
}
```

**影响**：客户端需要适配实际格式。
